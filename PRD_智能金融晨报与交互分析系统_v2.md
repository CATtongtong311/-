# 智能金融晨报与交互分析系统 — 产品需求文档（PRD）

> **版本**：v2.1  
> **更新日期**：2026-04-28  
> **目标用户**：A股/港股短线投资者，多板块跟踪（半导体、AI算力、液冷、商业航天等）

---

## 一、项目概述

### 1.1 目标
构建一套基于飞书（Feishu）平台的智能金融助手，具备两大核心能力：
1. **定时晨报推送**：每日开盘前（08:30）自动推送个性化金融晨报；
2. **交互式股票诊断**：用户可在飞书群内 @机器人 输入股票代码，实时获取当日详情与交易建议。

### 1.2 核心原则
> **"3分钟读完，知道今天该看什么；30秒交互，知道这只票能不能买。"**

- 不堆砌信息，只输出可指导交易决策的要点；
- 所有输出必须结构化、可验证、带红标风险提示；
- 内部由多个专业变体 Agent 协作，模拟投研团队分工。

## 二、功能需求

### 2.1 功能一：定时晨报推送（Morning Briefing）

#### 2.1.1 触发机制
| 配置项 | 说明 |
|--------|------|
| 触发时间 | 每天 08:30（北京时间），周末及法定节假日不跳过 |
| 预拉取时间 | 19:00 启动数据预拉取，保存到本地 JSON 缓存，次日直接使用 |
| 失败重试 | 08:25 若未生成完毕，发送延迟警告并附带"数据截止 08:25" |
| 推送渠道 | 飞书自建应用机器人（WebSocket 长连接 + 消息卡片） |
| 目标群聊 | 通过 `FEISHU_DEFAULT_CHAT_ID` 配置默认推送群 |

#### 2.1.3 数据预拉取与缓存策略
- **每晚 19:00** 自动遍历 `portfolio.md` 中的持仓股，拉取行情与新闻到本地缓存；
- **缓存结构**：`data/cache/quotes/{code}_{date}.json`、`data/cache/news/{code}_{date}.json`、`data/cache/market/{date}.json`；
- **TTL**：默认 24 小时，次日诊股和晨报优先命中缓存，秒级响应；
- **过期清理**：每日 03:00 自动清理 7 天前的过期缓存文件；
- **预拉取通知**：完成后发送飞书文本汇总（成功/失败数、全球市场状态）。

#### 2.1.2 晨报内容结构（6大固定模块）

**模块 1：隔夜全球市场速览（宏观锚定）**
- 美股三大指数涨跌幅 + 纳斯达克中国金龙指数；
- 恒生指数期货夜盘表现（直接影响港股开盘）；
- 美元指数、离岸人民币、10Y 美债收益率；
- **一句话定性**："偏多/偏空/中性，关注 X 板块开盘承接力"。

**模块 2：持仓个股重大事件（持仓警报）**
- 自动匹配用户自选股列表（如：亚翔集成、兆易创新、神剑股份）；
- Overnight 公告：业绩预告、增减持、大宗交易、龙虎榜后续；
- 行业政策/海外映射：英伟达 GTC 进展、TSMC 法说会指引；
- **红标规则**：若触发成本价 ±5% 的催化因素，标题加 🔴。

**模块 3：今日重点板块前瞻（机会挖掘）**
- 基于昨日涨停梯队 + 隔夜舆情热度，输出 2-3 个重点板块；
- 每个板块给出：催化事件、核心标的（1-2 只龙头）、量能预判。

**模块 4：龙虎榜与机构动向（资金验证）**
- 昨日龙虎榜净买入前 5 的板块分布；
- 机构专用席位大额买卖的个股（与用户自选交叉匹配）；
- **异常信号**：如"某液冷股昨日机构净买 1.2 亿，但今日无利好公告，关注是否提前知道订单"。

**模块 5：今日关键时间节点（交易提醒）**
- A股/港股今日重要财报披露（自选 + 行业龙头）；
- 股东大会、限售解禁、北向资金休市提醒；
- **持仓专属提醒**：如"亚翔集成 Q1 预报窗口期，注意 15:00 后是否有预披露"。

**模块 6：大盘技术位与策略建议（操作框架）**
- 上证指数/恒生指数关键支撑/压力位；
- 量能预判：基于近 5 日成交额均线，判断今日是否放量；
- **策略建议**（三选一）：进攻型 / 防御型 / 观望型，给出具体支撑压力位与止损线。

### 2.2 功能二：飞书交互式股票诊断（Interactive Query）

#### 2.2.1 交互方式
用户在飞书群聊或单聊中，通过以下方式触发：

```
@机器人 002709              # A股代码
@机器人 天赐材料             # 中文名称（仅限 portfolio.md 中的持仓股）
@机器人 0981.HK             # 港股代码
```

#### 2.2.2 机器人识别规则
- **A股代码**：6 位数字（首位 0/3/6），如 `002709`；
- **港股代码**：1-5 位数字 + `.HK`，如 `0981.HK`；
- **中文名称**：仅限 `portfolio.md` 中已维护的持仓股名称，如"天赐材料"会自动匹配到 `002709`；
- 若输入无效或名称未在持仓中匹配到，返回："请输入有效的股票代码或持仓中的股票名称"；
- 若只 @机器人未输入任何内容，返回："我在！请输入股票代码或名称，例如：002709"；
- 行情数据优先命中本地缓存，无缓存时自动降级调用 API。

#### 2.2.3 实时诊断输出结构（飞书消息卡片）

当用户查询某只股票时，机器人返回一张**飞书交互卡片**，包含以下区块：

| 卡片区块 | 内容 |
|---------|------|
| **Header** | 股票名称 + 代码 + 当前价 + 涨跌幅（颜色：红/绿/灰） |
| **技术速览** | 当日关键价位：开盘价、最高/最低、成交量 vs 5日均量、换手率 |
| **资金信号** | 龙虎榜/机构流向（如有）、北向资金持股变化、主力资金净流入 |
| **多 Agent 会诊** | 汇总 4 个变体 Agent 的结论（见第 3 章），以标签形式展示 |
| **操作建议** | 综合评分 + 一句话策略（如"短线观望，等待回踩 20 日线确认"） |
| **操作按钮** | [查看分时图] [加入自选] [深度研报]（占位，后续迭代） |

#### 2.2.4 时效性要求
- 从用户发送消息到返回卡片，**全程不超过 15 秒**；
- 行情数据允许 3 分钟延迟（非 L2 实时），但必须在卡片底部标注"数据截止 HH:MM"。

## 三、多 Agent 变体架构（Multi-Agent Orchestration）

系统内部不再由单一 Agent 处理所有任务，而是由 **Master Agent（主控）** 动态派生、调度多个 **Sub-Agent（变体专家）**，模拟真实投研团队的分工协作。

### 3.1 架构图

```
┌─────────────────────────────────────────────┐
│              用户输入层                        │
│  （飞书消息 / 定时 Cron / 后台管理）            │
└─────────────────────────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────────┐
│           Master Agent（主控/调度）             │
│  • 意图识别：晨报 / 个股诊断 / 板块扫描         │
│  • 任务拆解：拆分为子任务并选择 Agent 类型      │
│  • 结果聚合：加权汇总，生成最终输出             │
│  • 冲突仲裁：当 Sub-Agent 结论矛盾时裁决        │
└─────────────────────────────────────────────┘
                    │
        ┌───────────┼───────────┐
        ▼           ▼           ▼           ▼
┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐
│  Agent-  │ │  Agent-  │ │  Agent-  │ │  Agent-  │
│  Tech    │ │  Funda   │ │  Quant   │ │  Senti   │
│  技术派  │ │  基本面派│ │  量化派  │ │  情绪派  │
└──────────┘ └──────────┘ └──────────┘ └──────────┘
```

### 3.2 变体 Agent 定义与职责

#### Agent-Tech（技术派分析师）
- **专长**：K线形态、均线系统、量能分析、支撑压力位、龙虎榜席位技术解读；
- **输入**：当日分时/日K数据、历史均线、换手率、龙虎榜买卖席位；
- **输出**：`{trend_signal, key_levels, volume_comment, entry_exit_zones}`；
- **提示词风格**："你是一名激进的短线技术派，只看图和量，不谈基本面。给出明确的支撑/压力与突破判断。"

#### Agent-Funda（基本面/事件派分析师）
- **专长**：财报解读、行业政策、订单公告、海外映射（NVIDIA/TSMC）、产业链验证；
- **输入**：F10 资料、最新公告、行业新闻、券商研报摘要；
- **输出**：`{catalyst_events, risk_flags, earnings_quality, sector_position}`；
- **提示词风格**："你是一名行业研究员，关注业绩真空期、预披露规则、订单兑现度。所有结论必须标注信息来源与披露时间。"

#### Agent-Quant（量化/统计派分析师）
- **专长**：波动率统计、资金流向建模、板块轮动概率、历史回测相似形态；
- **输入**：近 20 日量价数据、板块资金流、北向/融资余额变化；
- **输出**：`{probability_bull, volatility_forecast, money_flow_score, mean_reversion_risk}`；
- **提示词风格**："你是一名量化分析师，所有结论必须带概率数字（如'上涨概率 62%'），禁止模糊表述。"

#### Agent-Senti（情绪/舆情派分析师）
- **专长**：舆情热度、股吧/雪球情绪、涨停梯队分析、题材持续性判断；
- **输入**：社交媒体热度指数、昨日涨停今日溢价率、连板高度、题材发酵度；
- **输出**：`{hype_level, sustainability_score, retail_greed_index, contrarian_signal}`；
- **提示词风格**："你是一名市场情绪猎手，判断题材是否处于加速/分歧/退潮期。当所有人都在讨论时，给出反向警告。"

### 3.3 Master Agent 调度逻辑

#### 3.3.1 任务类型与 Agent 组合映射

| 任务类型 | 调用的 Sub-Agent 组合 | 聚合权重 |
|---------|---------------------|---------|
| **定时晨报** | Tech(30%) + Funda(30%) + Senti(25%) + Quant(15%) | 主看宏观与板块 |
| **个股诊断（短线）** | Tech(40%) + Senti(30%) + Quant(20%) + Funda(10%) | 主看图+情绪 |
| **个股诊断（中线）** | Funda(40%) + Tech(25%) + Quant(25%) + Senti(10%) | 主看基本面 |
| **板块扫描** | Senti(35%) + Tech(30%) + Quant(25%) + Funda(10%) | 主看情绪与资金 |
| **持仓警报** | Funda(40%) + Tech(30%) + Quant(20%) + Senti(10%) | 事件驱动优先 |

#### 3.3.2 冲突仲裁规则
当多个 Agent 结论矛盾时（如 Tech 看多 vs Senti 看空）：
1. **时间框架优先**：短线任务以 Tech + Senti 为准；中线任务以 Funda + Quant 为准；
2. **置信度加权**：若某 Agent 给出概率 >80% 且论据充分（带具体数据），提升其权重 10%；
3. **人类兜底**：若分歧度 >60%（如 2:2 平局），Master Agent 输出"多派观点并列，建议观望"，不强行统一。

#### 3.3.3 动态提示词派生（Meta-Prompting）
Master Agent 不直接处理分析，而是为每个 Sub-Agent **动态生成专属提示词**：

```python
# 示例: Master Agent 为 Tech Agent 生成的动态提示词
tech_prompt = generate_prompt(
    role="激进的短线技术派分析师",
    sector="半导体设备",
    symbol="亚翔集成",
    cost_basis="207",
    focus_metrics="量能、换手率、龙虎榜机构流向",
    tools=["量能计算模块", "均线分析器", "龙虎榜席位追踪"],
    output_schema="JSON: {trend, support, resistance, volume_verdict, seat_analysis}"
)
```

## 四、数据源与接口清单

### 4.1 数据类型与来源

| 数据类型 | 实际源 | 获取方式 | 降级链路 |
|---------|--------|---------|---------|
| A股行情 | Tushare Pro → AKShare（东财/新浪）→ iTick | API | 三级自动降级 |
| 港股行情 | AKShare（新浪）→ iTick | API | 两级自动降级 |
| 全球市场 | AKShare（新浪）→ iTick | API | 两级自动降级 |
| A股公告/新闻 | AKShare（东财） | API | 单源 |
| 持仓管理 | `portfolio.md` 本地文件 | 文件读取 | 每次查询实时解析 |

### 4.2 多源降级策略

系统采用 **本地缓存优先 + 三级自动降级** 机制：

```
用户查询
    │
    ▼
本地 JSON 缓存（24h TTL）→ 命中则直接返回
    │
    ▼ 未命中或 force_refresh
Tushare Pro（主源）→ 额度充足且数据正常则返回
    │
    ▼ 无权限/额度不足/数据异常
AKShare（备用源）→ 东财个股信息 + 新浪历史日线
    │
    ▼ 接口失效/代理异常
iTick（兜底源）→ 免费版 5次/分钟，支持 A股/港股/美股指数
    │
    ▼ 全部失败
返回 "数据暂不可用"，飞书提示用户
```

### 4.3 本地缓存策略
- **缓存介质**：JSON 文件（`data/cache/`），无需 Redis；
- **缓存结构**：
  - `quotes/{code}_{date}.json` — 个股行情（OHLCV、涨跌幅、来源）
  - `news/{code}_{date}.json` — 个股新闻（标题、摘要、时间、来源）
  - `market/{date}.json` — 全球市场快照（美股三大指数、美债等）
- **TTL**：默认 24 小时，诊股和晨报优先命中缓存，秒级响应；
- **预拉取**：每晚 19:00 遍历 `portfolio.md` 持仓股，force_refresh 刷新全部缓存；
- **过期清理**：每日 03:00 自动清理 7 天前的缓存文件；
- **AI 快速查询**：提供 `get_cached_quote_as_text()` 方法，将缓存数据格式化为 Markdown 文本供 Claude Code CLI 直接嵌入 Prompt。

## 五、飞书部署方案

### 5.1 机器人接入方式
采用 **飞书自建应用 + 群机器人 + 长连接（WebSocket）** 方案，支持双向交互 [^6^][^7^]。

#### 5.1.1 创建飞书应用
1. 登录 [飞书开放平台](https://open.feishu.cn)；
2. 创建"企业自建应用"，启用"机器人"能力；
3. 权限配置（批量导入）：
   ```json
   {
     "scopes": {
       "tenant": [
         "im:message:send_as_bot",
         "im:message.group_at_msg:readonly",
         "im:chat.members:read"
       ]
     }
   }
   ```
4. 发布到目标群聊，获取 **Webhook URL** 与 **App ID / App Secret** [^3^][^5^]。

#### 5.1.2 消息接收模式选择
| 模式 | 适用场景 | 说明 |
|------|---------|------|
| **长连接（WebSocket）** | 开发/测试阶段、无公网 IP | SDK 自动维护连接，无需内网穿透，推荐优先使用 [^6^] |
| **Webhook 回调** | 生产环境、有公网服务器 | 需配置公网 URL、验证 Token、IP 白名单 [^3^] |

#### 5.1.3 消息卡片格式
晨报与诊断结果均使用 **飞书 Interactive Card（消息卡片）** 推送，支持：
- 标题颜色区分（红/绿/橙/灰）；
- 分栏布局（多 Agent 结论并列展示）；
- 按钮交互（后续可扩展"加入自选"、"查看深度报告"等）[^2^][^15^]。

### 5.2 实际部署架构（单进程 + 本地缓存）


**关键配置说明**：
- `FEISHU_DEFAULT_CHAT_ID`：在群里 @机器人后，从服务日志中获取 `chat_id` 填入；
- `TUSHARE_ITICK_TOKEN`：iTick 免费版 token，作为 AKShare 失败后的第三级降级；
- 所有敏感信息通过环境变量注入，`config/settings.py` 使用 `pydantic-settings` 自动读取。

## 六、非功能需求

### 6.1 性能指标（实际达成）
| 指标 | 目标值 | 实际值 |
|------|--------|--------|
| 晨报生成耗时 | < 30 分钟 | ~2 分钟（缓存命中后 AI 生成） |
| 个股诊断响应 | < 15 秒 | ~3-5 秒（缓存命中）/ ~30-180 秒（需 AI 分析） |
| 并发查询 | 10 用户 | 单线程顺序处理（个人 demo，暂不支持并发） |
| 服务可用性 | 99.5% | WebSocket 自动重连（10 次指数退避） |
| 数据延迟 | 3 分钟 | Tushare 3 分钟延迟 / AKShare 实时 |
| 缓存响应 | — | 本地 JSON 缓存，秒级读取 |

### 6.2 安全与合规
- **密钥管理**：所有 API Key、App Secret 通过环境变量注入，禁止硬编码；
- **数据隐私**：用户持仓成本等敏感数据仅保存在本地 Redis/DB，不上传至 LLM 训练；
- **免责声明**：每条消息卡片底部固定添加"AI生成内容仅供参考，不构成投资建议"。

### 6.3 扩展性
- 新增 Agent 变体时，只需在 Master Agent 的调度表中注册权重，无需改动底层架构；
- 新增数据源时，通过统一的数据接口层（Data Adapter）接入，不影响分析层。

### 附录 A：Master Agent 系统提示词（核心 Meta-Prompt）

```markdown
你是 Master Agent（主控调度员），负责管理一个 4 人投研团队的输出。
你的任务不是直接分析股票，而是：
1. 识别用户意图（晨报 / 个股诊断 / 板块扫描）；
2. 根据任务类型，为每个 Sub-Agent 动态生成专属提示词；
3. 接收 4 个 Agent 的 JSON 输出，按预设权重加权聚合；
4. 当结论冲突时，依据"时间框架优先 + 置信度加权"规则仲裁；
5. 输出最终结构化结论（Markdown / JSON）。

可用 Sub-Agent 列表：
- Agent-Tech：技术派，专精 K线/量能/龙虎榜
- Agent-Funda：基本面派，专精财报/政策/产业链
- Agent-Quant：量化派，专精概率/统计/资金流
- Agent-Senti：情绪派，专精舆情/题材/涨停梯队

调度权重表（已内置，按任务类型自动选择）。
禁止在最终输出中暴露内部 Agent 的权重分配细节。
```
