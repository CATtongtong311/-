# Phase 5: Kimi Agent 网页直出晨报改造方案

> **目标**: 将晨报生成主体从本地 Claude Code CLI 迁移至 Kimi Agent 网页 (https://www.kimi.com/agent)，由 Kimi 直接输出完整 Markdown 晨报，本地仅负责数据预注入、MD 抓取、入库与飞书发送。
>
> **Kimi 回复超时**: 20 分钟（1200 秒）
>
> **文档状态**: 已完成 | **创建日期**: 2026-04-28 | **完成日期**: 2026-04-30

---

## 1. 改造目标与范围

### 1.1 核心变更

| 模块 | 改造前 | 改造后 |
|------|--------|--------|
| **晨报生成引擎** | 本地 Claude Code CLI (`claude -p`) | Kimi Agent 网页 (https://www.kimi.com/agent) |
| **任务一：金融新闻** | Tushare `major_news` + AKShare `stock_news_em` | Kimi Agent 联网搜索直出 |
| **任务二：板块异动** | 无（仅关注板块名称） | Kimi Agent 联网搜索直出 |
| **市场情绪判断** | 无 | Kimi Agent 基于新闻与板块分析直出 |
| **今日关注提示** | 无 | Kimi Agent 基于综合研判直出 |
| **全球市场数据** | Tushare → AKShare → iTick 三级降级 | 本地 API 获取后注入 Kimi Prompt |
| **持仓个股新闻** | Tushare → AKShare | 本地 API 获取后注入 Kimi Prompt |
| **诊股功能** | Claude Code CLI | **保持不变** |

### 1.2 不变的功能

- 飞书 WebSocket 连接与消息交互
- 诊股分析（四维度 + 结构化输出）
- 三级数据降级（Tushare → AKShare → iTick）—— 作为 Kimi 失败时的数据源备用
- 本地 JSON 缓存（24h TTL）
- 持仓 ±5% 警示检测
- 飞书卡片发送与免责声明注入
- 7 天数据自动清理

### 5.1 KimiAgentBrowser (`src/llm/kimi_agent_browser.py`)

```python
class KimiAgentBrowser:
    """通过 Playwright 自动化访问 https://www.kimi.com/agent 生成晨报。
    
    超时配置: 20 分钟 (1200 秒)
    """
    
    DEFAULT_TIMEOUT = 1200  # 20 分钟
    KIMI_URL = "https://www.kimi.com/agent"
    
    def __init__(self, cookie_path: str = "data/kimi_cookies.json"):
        self.cookie_path = Path(cookie_path)
        self.playwright = None
        self.browser = None
        self.context = None
        self.page = None
    
    async def _init_browser(self) -> None:
        """初始化 Playwright，使用 Chromium。"""
        
    async def _ensure_login(self) -> bool:
        """检查登录态，Cookie 失效时返回 False（触发降级）。"""
        
    async def _load_cookies(self) -> None:
        """从本地加载 Cookie。"""
        
    async def _save_cookies(self) -> None:
        """保存当前 Cookie 到本地。"""
        
    async def send_prompt(self, prompt: str, timeout: int = DEFAULT_TIMEOUT) -> str:
        """发送 Prompt 到 Kimi Agent，等待完整回复。
        
        Args:
            prompt: 完整的超级 Prompt（含注入数据）
            timeout: 最大等待时间，默认 1200 秒
            
        Returns:
            Kimi 生成的完整 Markdown 文本
            
        Raises:
            KimiTimeoutError: 超过 timeout 仍未完成生成
            KimiLoginError: 登录态失效
            KimiFormatError: 返回内容格式异常
        """
        # 1. 访问 Kimi Agent 页面
        # 2. 定位输入框，输入 Prompt
        # 3. 点击发送
        # 4. 轮询检测生成完成（每5秒检查 DOM）
        #    - 检测标志: "重新生成"按钮可见
        #    - 或: 输出内容 30 秒无变化
        #    - 或: 出现错误提示
        # 5. 抓取完整回复文本
        # 6. 关闭页面（不关闭浏览器，复用连接）
        
    async def generate_report(self, input_data: dict) -> dict:
        """完整的晨报生成入口。
        
        Args:
            input_data: {
                "date": "2026-04-28",
                "yesterday": "2026-04-27",
                "market_snapshot": {...},
                "holdings_news": {...},
                "portfolio": {...}
            }
            
        Returns:
            {
                "success": True,
                "markdown": "# 晨报 2026-04-28\n...",
                "sentiment": {"mood": "谨慎", "score": 42},
                "raw_length": 3500,
                "elapsed_sec": 680
            }
        """
        
    async def close(self) -> None:
        """关闭浏览器，释放资源。"""
```

### 5.3 DOM 完成检测逻辑

```python
async def _wait_for_completion(self, timeout: int = 1200) -> bool:
    """轮询检测 Kimi 回复是否生成完成。"""
    start = time.time()
    last_content = ""
    stale_count = 0  # 内容未变化计数
    
    while time.time() - start < timeout:
        await asyncio.sleep(5)  # 每 5 秒检测一次
        
        # 获取当前回复内容
        current = await self._extract_reply_text()
        
        # 检测完成标志1: "重新生成"按钮出现
        regen_visible = await self._is_regenerate_button_visible()
        if regen_visible and len(current) > 500:
            return True
            
        # 检测完成标志2: 内容 30 秒无变化（连续6次轮询相同）
        if current == last_content:
            stale_count += 1
            if stale_count >= 6 and len(current) > 500:  # 30秒无变化
                return True
        else:
            stale_count = 0
            
        # 检测错误标志
        if await self._is_error_visible():
            raise KimiFormatError("Kimi 返回错误提示")
            
        last_content = current
    
    raise KimiTimeoutError(f"超过 {timeout} 秒仍未完成生成")
```

### 8.1 新任务时间表

```python
class SchedulerManager:
    def start(self) -> None:
        # 07:30: 数据预准备（全球市场 + 持仓新闻）
        self.scheduler.add_job(
            self._run_pre_fetch_report_data,

        # 07:35: Kimi Agent 晨报生成（预留20分钟）
        self.scheduler.add_job(
            self._run_kimi_generate,
            trigger=CronTrigger(hour=7, minute=35),
            id="kimi_generate",
        )
        
        # 08:00: 晨报飞书发送（Kimi 生成完成后）
        self.scheduler.add_job(
            self._run_morning_report_send,
            trigger=CronTrigger(hour=8, minute=0),
            id="morning_report_send",
        )
        
        # 19:00: 晚间数据预拉取（原有，不变）
        self.scheduler.add_job(
            self._run_pre_fetch,
            trigger=CronTrigger(hour=19, minute=0),
            id="pre_fetch_data",
        )
        
        # 03:00: 过期数据清理（原有，不变）
        self.scheduler.add_job(
            self._run_cleanup,
            trigger=CronTrigger(hour=3, minute=0),
            id="cleanup_reports",
        )
```

### 8.2 任务间依赖说明

三个晨报相关任务通过**本地文件状态**进行隐性依赖：

```
pre_fetch_report_data (07:30)
  └→ 写入 data/cache/report_input/{date}.json

kimi_generate (07:35)
  └→ 读取 report_input/{date}.json
  └→ 写入 data/cache/kimi_report/{date}.md
  └→ 写入 SQLite kimi_raw_data 表

morning_report_send (08:00)
  └→ 读取 kimi_report/{date}.md
  └→ 发送飞书卡片
```

若 07:35 任务失败（超时/异常），08:00 任务检测到无 `kimi_report/{date}.md`，自动触发降级到 Claude 生成。

## 12. 降级策略详解

### 12.1 三级降级链路

```
第一级：Kimi Agent 网页生成（主路径）
    ├→ 成功：直接入库发送
    ├→ 超时（>1200s）：降级到第二级
    ├→ 登录失败：降级到第二级
    └→ 格式错误：降级到第二级

第二级：本地 Claude Code CLI 生成（保留原代码）
    ├→ 成功：source="claude_fallback"
    └→ 失败：降级到第三级

第三级：简化文本通知（兜底）
    └→ 发送纯文本："晨报生成失败，请手动查看市场数据"
```

### 12.2 降级触发条件与日志

| 场景 | 触发条件 | 降级动作 | 飞书通知 |
|------|----------|----------|----------|
| Kimi 超时 | `elapsed > 1200s` | 调用 `generate_fallback()` | "⚠️ Kimi生成超时(20min)，已启用Claude备用方案" |
| Kimi 登录失效 | Cookie 过期，重定向到登录页 | 调用 `generate_fallback()` | "⚠️ Kimi登录态失效，已启用Claude备用方案，请执行 `python -m src.llm.kimi_agent_browser --login` 刷新Cookie" |

## 15. 实际执行差异（2026-04-30 补充）

以下是在实际开发过程中，超出原计划或未在计划中明确预见的关键调整：

### 15.1 固定等待策略替代纯轮询检测（重大调整）

- **原计划**：轮询检测 DOM 状态，每 5 秒检查"重新生成"按钮是否出现，内容 30 秒无变化则判定完成。
- **实际执行**：Kimi 左侧思考过程会持续滚动，轮询期间操作导致提取内容被污染。改为 `fixed_wait_sec=420`（7 分钟）固定等待，确保思考过程滚动完毕后再进行提取/下载操作。
- **影响**：`KimiAgentBrowser.send_prompt` 新增 `fixed_wait_sec` 参数，原有轮询逻辑保留作为兜底。

### 15.2 文本提取三重去污染

- **原计划**：仅通过 DOM 选择器提取 Kimi 返回的 Markdown 文本。
- **实际执行**：发现提取文本混入了用户 Prompt 和思考过程，新增三重过滤：
  1. **坐标过滤**：排除左侧边栏（x < 260）和右侧 Computer 面板（x > 1000）的文本
  2. **思考过程过滤**：JS 端排除包含 `thinkingKeywords`（"深度思考"、"反思"等）的文本节点
  3. **Prompt 排除**：传入 `exclude_text` 参数排除 prompt 前 100 字符

### 15.3 下载事件可靠性修复

- **原计划**：`page.on("download")` 回调中直接保存文件。
- **实际执行**：Playwright 的 download 事件为异步回调，不等待完成就关闭浏览器导致文件未保存。修复方案：
  - 改为 `asyncio.create_task(_handle())` 创建后台任务
  - 关闭前 `await asyncio.gather(*pending_download_tasks, return_exceptions=True)` 等待所有下载完成

### 15.4 飞书卡片格式大幅优化（超出原计划范围）

- **原计划**：仅实现情绪评级颜色映射（乐观绿/中性蓝/谨慎红）。
- **实际执行**：卡片渲染做了全面重构，超出原计划 20 行的预估：
  - 章节拆分渲染：一级标题 `heading` + `blue`，二级标题 `normal` + `grey`
  - 章节 emoji 图标映射（🌍 🇨🇳 📜 🏭 🔭 📊 💼 🎯）
  - 智能 `hr` 分割线策略：一级无内容且下一个是二级时不加 hr；二级之间不加 hr
  - 制表符表格自动转换为标准 Markdown 表格（`| col1 | col2 |`）
  - "数据缺失"内容转为 `> ⚠️` 引用块（灰色背景）
  - 情绪标签在卡片顶部醒目显示（`heading` 字号 + 情绪对应颜色）

### 15.5 Sentiment 正则扩展

- **原计划**：仅支持 Markdown 加粗格式提取情绪（`**情绪评级**：乐观`）。
- **实际执行**：Kimi 返回的表格使用制表符分隔（`情绪评级\t乐观`），正则扩展为同时支持两种格式：
  - `**情绪评级**[:：]情绪` 和 `情绪评级[:：\s]+情绪`
  - `**情绪得分**[:：]数字` 和 `情绪得分[:：\s]+数字`

### 15.6 持久化上下文复用登录态

- **原计划**：每次启动新浏览器上下文，从 Cookie 文件加载。
- **实际执行**：改为 `playwright.chromium.launch_persistent_context()` 使用用户数据目录，Cookie 自动持久化到磁盘，无需手动加载/保存。

### 15.7 卡片标签兼容性修复

- **未预见问题**：飞书 Card JSON 2.0 不支持 `note` 和 `remark` 标签，导致发送失败（错误码 200861、200621）。
- **修复**：全部改为 `div` + `plain_text` + `text_size="notation"` 替代。
