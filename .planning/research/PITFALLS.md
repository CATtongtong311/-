# 智能金融晨报与交互分析系统 — 常见陷阱

**Domain:** 飞书金融数据助手 / 个人投资者工具
**Researched:** 2026-04-25
**Confidence:** MEDIUM（基于社区实践、GitHub Issue、官方文档及行业经验，部分数据为2025年信息）

---

## Critical Pitfalls

### Pitfall 1: 飞书 WebSocket 长连接"假存活"——机器人看似在线实则收不到消息

**What goes wrong:**
机器人进程仍在运行，日志无报错，但用户@机器人后无任何响应。重启后恢复正常，数小时后又出现同样问题。这是飞书 WebSocket 机器人最常见的故障模式。

**Why it happens:**
- WebSocket 连接在网络抖动后进入"僵尸状态"——TCP 连接未断开，但飞书服务器已不再向该连接推送事件
- 心跳机制（Ping/Pong）配置不当或未被正确处理
- 未启用飞书开放平台的"使用长连接接收事件"模式，或事件订阅配置遗漏 `im.message.receive_v1`
- 应用未发布版本，仅测试用户可用，导致群聊中机器人无响应

**How to avoid:**
- 实现双重健康检查：（1）WebSocket 层 Ping/Pong 超时检测；（2）业务层定时自检（如每5分钟向自己发送测试消息验证通路）
- 配置连接断开后指数退避重连（首次1秒，最大60秒）
- 在飞书开放平台确认：事件订阅 -> 使用长连接接收事件 -> 已添加 `im.message.receive_v1`
- 应用必须发布版本，群聊中需@机器人才能触发（单聊可直接发送）
- 使用 `replace_existing=True` 等幂等配置，防止重连后重复注册任务

**Warning signs:**
- 日志中长时间无 `im.message.receive_v1` 事件记录
- 飞书开放平台"机器人互动"统计中消息接收数突然归零
- 手动重启后立刻恢复正常

**Phase to address:** Phase 1（飞书机器人接入）

### Pitfall 2: 飞书消息卡片 30KB 超限导致晨报推送失败

**What goes wrong:**
晨报内容较长时，飞书 API 返回错误码 `230025` 或 `19036`，消息发送失败。开发者本地测试正常（内容较短），但真实晨报生成后因数据丰富而超限。

**Why it happens:**
- 飞书卡片消息和富文本消息最大限制为 **30KB**（UTF-8 编码字节数，非字符数）
- 样式标签会膨胀实际大小：Markdown 样式、嵌套组件、颜色属性等都会占用字节
- 换行符 `\n` 计为2个字符，空格也计入限制
- 卡片 JSON 元素数量最多 **200 个**
- 文本消息最长 **500 字符**，Markdown 最长 **5000 字符**

**How to avoid:**
- 发送前预检查卡片大小：`len(json.dumps(card, ensure_ascii=False).encode('utf-8')) / 1024`
- 超过 25KB 时自动触发内容裁剪策略：优先保留核心数据（持仓警示、今日重点），折叠次要内容（全球市场详情）
- 使用 `plain_text` 替代复杂 `lark_md` 减少样式开销
- 长文本分段发送，或使用"查看详情"按钮链接到完整报告
- 对常用卡片模板进行本地缓存，避免重复计算

**Warning signs:**
- 本地单股测试正常，多持仓组合测试时偶发失败
- 错误日志中出现 `230025`、`19036` 或 `300302`
- 卡片更新时提示"更新卡片实体配置超限"

**Phase to address:** Phase 1（晨报推送功能）

### Pitfall 3: Tushare 免费额度耗尽导致数据获取中断

**What goes wrong:**
系统运行一段时间后，所有 Tushare 接口返回空数据或报错，手动执行也失败。次日自动恢复，但当日晨报/诊股功能完全不可用。

**Why it happens:**
- Tushare 免费用户每日调用额度约为 **100-500 次**（不同账户略有差异）
- `pro_bar` 接口获取分钟级数据需要 **600 积分以上** 才能使用
- 无数据返回时仍会计入调用次数（GitHub Issue #1703 已确认此问题）
- 股票代码未带交易所后缀（如 `000625` 而非 `000625.SZ`）会导致查询失败但扣积分
- 高频调用易触发反爬虫机制，导致 Token 临时冻结

**How to avoid:**
- 实现调用计数器，接近额度上限时自动切换备用数据源（AkShare、新浪财经）
- 所有股票代码必须规范化：`code + '.SZ'`（深圳）或 `code + '.SH'`（上海）
- 批量获取数据：使用 `daily` 接口一次拉取多只股票，减少 API 调用次数
- 本地 SQLite 缓存历史数据，避免重复调用
- 避开交易时段高峰期（9:30-11:30, 13:00-15:00），建议在盘后或非交易时间预取数据
- 每次调用前检查返回是否为 `None`，为 `None` 时记录日志但不重复重试

**Warning signs:**
- 接口返回 `None` 但调用次数增加
- 出现 `API call frequency limit exceeded` 错误
- 交易时段数据获取成功率明显低于非交易时段

**Phase to address:** Phase 1（数据层搭建）

### Pitfall 4: 金融数据质量陷阱——停牌、复权、空值导致分析失真

**What goes wrong:**
诊股报告中的技术指标计算异常（如 RSI 为负值、均线断裂），或持仓股停牌后仍显示"可买入"建议。用户发现数据与交易软件不一致，信任度骤降。

**Why it happens:**
- **停牌数据缺失**：停牌期间无行情数据，直接计算指标会导致 NaN 或错误值
- **复权方式不一致**：不同数据源前复权/后复权算法不同，价格差异可达数个百分点
- **OHLC 逻辑错误**：数据源偶尔出现 `high < low` 或 `close > high` 的脏数据
- **未来函数**：使用当日收盘后数据生成"盘中"建议（如用收盘价计算盘中买卖点）
- **除权除息日跳空**：未复权数据在除权日出现巨大跳空，误导趋势判断

**How to avoid:**
- 数据获取后强制校验：检查 `high >= low`、`close 在 [low, high] 范围内`、`volume >= 0`
- 识别停牌状态（成交量为0或特定标记），停牌股票在诊股报告中明确标注"停牌中"
- 统一使用**前复权**数据进行技术分析，实盘建议时注明"基于前复权数据"
- 缺失数据采用前向填充（`ffill`）保持连续性，但标记为"估算值"
- 所有指标计算使用**当日开盘前已确定的数据**（昨日及历史数据），绝不混入当日实时数据
- 多数据源交叉验证：Tushare 与新浪财经数据差异超过阈值时触发告警

**Warning signs:**
- 某只股票技术指标突然跳变（如 RSI 从 60 变为 5）
- 除权除息日后均线出现断崖式变化
- 诊股建议与主流交易软件（同花顺、东方财富）明显矛盾

**Phase to address:** Phase 1（数据层搭建）

### Pitfall 5: LLM 金融幻觉——编造财报数据、错误匹配股票代码

**What goes wrong:**
诊股报告中出现"该公司 Q4 出口增长 200%"等虚构数据，或将股票代码与公司名称错误匹配（如把"中芯国际"的代码错配给"中兴通讯"）。用户据此操作可能产生实际损失。

**Why it happens:**
- LLM 训练数据有截止日期，对最新财报、公告、股价一无所知
- 推理型模型（如 DeepSeek-R1）幻觉率可达 **14.3%**，是通用模型的 4 倍以上
- 金融数据具有高度精确性要求，1% 的误差可能导致完全错误的投资结论
- 模型倾向于"合理推测"而非承认不知道，会编造看似合理的财务数据
- 股票简称相似时（如"中芯国际"vs"中兴通讯"），模型容易混淆

**How to avoid:**
- **强制 RAG 架构**：所有财务数据、股价、公告必须通过 API 实时获取，注入 LLM Prompt 作为上下文
- **明确指令**：在 System Prompt 中写入"如果数据不在提供的上下文中，回答'数据暂不可用'，绝不要编造"
- **数值校验层**：LLM 输出中的数字必须与原始数据比对，差异超过阈值时拒绝输出并告警
- **股票代码锁定**：诊股时先将代码与公司名称绑定，LLM 分析全程使用代码而非名称引用
- **多模型交叉验证**：Claude 和 Kimi 分别生成分析，关键结论不一致时标记"需人工确认"
- 每条消息底部固定添加合规声明："AI生成内容仅供参考，不构成投资建议"

**Warning signs:**
- 报告中出现具体数字但无数据来源标注
- 建议与公开信息明显矛盾（如建议买入已停牌股票）
- 同一股票连续两次分析给出截然相反的结论

**Phase to address:** Phase 2（诊股交互功能）

### Pitfall 6: 定时任务静默失败——晨报未推送却无告警

**What goes wrong:**
某日开盘后发现没有收到晨报，查看日志发现定时任务未执行或执行失败但无通知。问题可能持续多日才被发现。

**Why it happens:**
- Python `asyncio.get_event_loop()` 在长时间运行后状态污染（Python 3.9+ 高危）
- Cron 环境变量缺失：`PATH` 仅加载极简路径，`python3` 命令找不到
- 相对路径陷阱：脚本内使用相对路径读取配置/数据文件，Cron 工作目录不确定
- 时区配置错误：服务器时区非 CST，任务在非预期时间触发或跳过
- 任务执行超时：晨报生成超过 Cron 超时限制，进程被强制终止
- 节假日未过滤：A股休市日（春节、国庆等）任务仍触发，浪费资源且推送无意义内容

**How to avoid:**
- 使用 APScheduler 替代系统 Cron，配置 `misfire_grace_time=300`、`coalesce=True`、`max_instances=1`
- 强制使用绝对路径和完整环境变量配置
- 实现交易日历过滤：调用 Tushare `trade_cal` 接口判断当日是否开市，非交易日跳过
- 任务执行后发送"心跳"消息到飞书：成功时发送简短确认，失败时发送告警
- 所有定时任务输出重定向到日志文件：`>> /var/log/finance-bot/cron.log 2>&1`
- 使用 `asyncio.to_thread()` 替代废弃的 `loop.run_in_executor()`

**Warning signs:**
- 连续多日日志中无晨报生成记录
- 飞书心跳消息中断超过1天
- 节假日收到无意义的空数据晨报

**Phase to address:** Phase 1（晨报推送功能）

### Pitfall 7: 个人 Demo 范围蔓延——从"晨报+诊股"膨胀为"量化交易平台"

**What goes wrong:**
项目最初目标明确（每日晨报 + 诊股查询），但在开发过程中不断加入新功能：回测系统、实时盯盘、多因子选股、自动化交易信号……最终代码量爆炸、Bug 丛生、核心功能始终不稳定，项目烂尾。

**Why it happens:**
- 金融数据领域诱惑极多：数据接口丰富、可视化效果炫酷、策略想法层出不穷
- 个人开发者容易陷入"完美主义拖延"——觉得核心功能"还不够好"，用新功能逃避交付
- 看到开源项目（如 stock-monitor）功能全面，产生攀比心理
- 缺乏外部反馈：个人项目没有产品经理或用户约束需求边界
- 技术自嗨：过度关注架构先进性（微服务、K8s、实时流处理），忽视业务价值

**How to avoid:**
- **严格遵循 PROJECT.md 中的 Out of Scope 清单**：Redis、复杂数据库、Webhook、L2行情、交易执行等明确不做的功能，任何情况下不讨论
- **第一期只做3个晨报模块**：隔夜全球市场、持仓个股事件、今日板块前瞻，其余模块明确标记为"二期"
- **设定硬截止日期**：例如"2周内必须跑通核心流程"，到期未完成的特性直接砍掉
- **每日自问**："今天做的工作是否让'开盘前3分钟读完晨报'或'@机器人30秒诊股'更快更稳？"
- 使用物理看板或 Markdown TODO 清单，任何新需求必须先写入"待评估"区，48小时后再决定是否纳入

**Warning signs:**
- 代码仓库中出现与核心功能无关的目录（如 `backtest/`、`realtime_monitor/`）
- 连续3天没有提交到核心功能分支
- 开始研究"以后可能用得上"的技术（如 Kafka、ClickHouse）
- 核心流程（晨报生成->推送）仍未端到端跑通

**Phase to address:** 所有阶段（贯穿整个项目周期）

### Pitfall 8: 多 Agent 过度工程化——7个 Agent 不如1个 Agent + 7个工具

**What goes wrong:**
诊股功能设计为"技术分析师 Agent + 基本面分析师 Agent + 情绪分析师 Agent + 综合诊断 Agent"的流水线，结果响应时间从5秒膨胀到15秒以上，各 Agent 输出矛盾（技术派说买入，基本面说卖出），调试困难。

**Why it happens:**
- 行业流行"Multi-Agent"概念，80%的多 Agent 系统被证实过度工程化
- 每增加一个 Agent，交互路径呈指数增长：2个 Agent 有3种交互，4个 Agent 有15种交互
- 协调开销（Token 浪费、延迟累积、状态同步）往往超过协作收益
- 金融分析任务虽然适合多 Agent，但**前提是任务边界清晰、可并行、有明确 SOP**——个人 Demo 阶段难以定义如此精确的边界

**How to avoid:**
- **第一期严格使用单 Agent 串行架构**：一个 LLM 调用，Prompt 中分章节要求分析技术面、基本面、情绪面，最后给出综合建议
- 将"多 Agent 会诊"降级为"单 Agent 多维度分析"——在 Prompt 中明确分区，而非物理拆分 Agent
- 如果确实需要拆分，遵循"5问题测试"：（1）上下文是否超过200K？（2）是否需要不同专业领域？（3）并行是否有价值？（4）是否有清晰交接点？（5）是否值得2-3倍成本？
- 通过 `function calling` / `tools` 让单 Agent 调用多个工具（查行情、查公告、查财务），而非创建多个 Agent

**Warning signs:**
- 诊股响应时间超过项目约束的15秒
- 同一股票两次查询给出逻辑矛盾的建议
- 调试时需要追踪3个以上 Agent 的输入输出
- 大部分开发时间在解决 Agent 间通信问题而非金融分析质量

**Phase to address:** Phase 2（诊股交互功能）

### Pitfall 9: 持仓 MD 文档与代码不同步——机器人读取过期持仓

**What goes wrong:**
用户已清仓某只股票，但机器人仍将其纳入晨报分析和持仓监控，导致推送无关信息甚至错误警示。用户修改 MD 文档格式后，解析失败，机器人完全无法读取持仓。

**Why it happens:**
- MD 文档无 Schema 约束，用户手动编辑时容易破坏格式（如遗漏分隔线、缩进错误）
- 机器人每次读取文档时未校验格式，解析失败时静默使用空持仓或上次缓存
- 无版本控制：用户不知道机器人读取的是哪个版本的持仓
- 持仓变更后，机器人未主动通知用户"已更新持仓上下文"

**How to avoid:**
- 定义严格的持仓 MD 格式模板，提供示例文件，关键字段（代码、成本价、持仓量）缺失时明确报错
- 读取 MD 后发送确认消息到飞书："已读取持仓：中芯国际(688981) x 1000股，成本 56.30"
- 实现简单的格式校验：正则表达式检查股票代码格式（6位数字）、成本价为数字
- 解析失败时发送告警卡片到飞书："持仓文档格式错误，请检查第X行"
- 在 SQLite 中缓存上次成功解析的持仓，MD 解析失败时回退到缓存并告警

**Warning signs:**
- 已清仓股票仍出现在晨报中
- 持仓成本价明显错误（如 0 或负数）
- 机器人对持仓查询无响应或返回空列表

**Phase to address:** Phase 1（持仓管理）

### Pitfall 10: 合规声明流于形式——投资建议被误信引发纠纷

**What goes wrong:**
用户将机器人的诊股建议视为专业投资意见进行操作，亏损后声称"AI误导"。虽然每条消息底部有免责声明，但建议内容过于具体（"建议买入，目标价 XX 元"），导致声明失效。

**Why it happens:**
- 中国证监会对 AI 投资建议有严格监管要求，个人 Demo 虽无持牌义务，但道德风险不可忽视
- LLM 输出语气过于肯定（"强烈建议买入"），用户容易忽略底部小字声明
- 缺乏风险提示分层：普通分析与涉及具体买卖点的建议应区分处理
- 项目文档中未明确"本项目仅为个人学习工具，不构成任何投资建议"

**How to avoid:**
- **声明前置**：将合规声明放在消息卡片**顶部**（而非底部），使用醒目的黄色/橙色背景
- **语气弱化**：Prompt 中要求 LLM 使用"从技术面看，当前处于...""基本面显示..."等客观描述，禁止使用"建议买入/卖出""目标价""必涨"等词汇
- **风险分级**：涉及具体价格预测时，额外添加"此分析基于历史数据，市场有风险，决策需谨慎"
- **README 和飞书机器人简介**中明确标注：本工具仅为个人学习演示，不提供投资建议
- 保留所有分析日志（输入数据、LLM 输出），以备争议时追溯

**Warning signs:**
- 用户截图分享机器人"建议"到投资群
- 诊股输出中出现具体目标价、止损位
- 用户反馈"按机器人说的买了，亏了"

**Phase to address:** Phase 2（诊股交互功能）

## Technical Debt Patterns

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| 持仓用 MD 文档而非数据库 | 无需设计表结构，手动编辑方便 | 格式易破坏，无事务一致性，并发编辑冲突 | MVP 阶段可接受，需在 Phase 2 增加校验层 |
| 内存 dict 缓存替代 Redis | 零部署，代码简单 | 进程重启缓存丢失，无法跨实例共享 | 个人 Demo 单进程场景可接受 |
| 单线程串行处理 | 无并发 Bug，逻辑简单 | 诊股响应慢，无法并行获取多源数据 | Phase 1 可接受，Phase 2 引入 asyncio |
| 硬编码股票板块映射 | 快速实现板块识别 | 板块调整时需改代码，维护困难 | MVP 阶段可接受，长期应接入动态分类接口 |
| 忽略节假日过滤 | 代码简单 | 休市日推送无意义内容，浪费 API 额度 | **不可接受**，必须在 Phase 1 实现 |
| 无数据质量校验直接传给 LLM | 代码量少 | 脏数据导致幻觉分析，信任崩塌 | **不可接受**，必须在 Phase 1 实现基础校验 |

## Integration Gotchas

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| 飞书 WebSocket | 未启用"长连接接收事件"模式，或事件订阅遗漏 | 确认 `im.message.receive_v1` 已订阅，应用已发布 |
| 飞书消息卡片 | JSON 超过 30KB 或元素超过 200 个 | 发送前预检查大小，超限时分片或折叠 |
| Tushare | 股票代码不带 `.SZ`/`.SH` 后缀 | 所有代码统一规范化，查询失败不重复重试 |
| Tushare | 无数据返回仍扣积分，导致额度快速耗尽 | 实现调用计数器，接近上限时切换备用源 |
| 新浪财经 | 非官方接口，IP 可能被封 | 作为备用源，主源失败时降级，不依赖为唯一数据源 |
| Claude API | 长文本超出上下文限制 | 精简 Prompt，只注入必要数据，使用 `claude-sonnet-4-6` 的长上下文 |
| Kimi API | 模型切换时输出格式不一致 | 定义统一的输出 Schema，两个模型共用同一解析逻辑 |

## Performance Traps

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| 每次诊股都实时拉取全部历史数据 | 响应时间 > 30 秒，Tushare 额度快速耗尽 | 本地 SQLite 缓存日线数据，只拉取最新1-2天 | 持仓超过10只股票时明显 |
| 晨报生成阻塞主线程 | 飞书机器人无响应，无法处理诊股请求 | 晨报生成在独立线程/进程中执行 | 用户同时@机器人和等待晨报时 |
| LLM 输出过长导致卡片超限 | 晨报推送失败，错误码 230025 | 内容分片，核心数据优先，详情折叠 | 持仓股超过5只或全球市场数据丰富时 |
| 无超时控制的 API 调用 | 诊股请求挂死，用户等待无反馈 | 所有外部调用设置 timeout（建议 10-15 秒） | 任何网络抖动时 |
| 同步阻塞式 WebSocket 处理 | 同时多个用户@机器人时排队等待 | 使用 asyncio 异步处理消息 | 群聊中多人同时使用时 |

## Security Mistakes

| Mistake | Risk | Prevention |
|---------|------|------------|
| Tushare Token / Claude API Key 硬编码在代码中 | 密钥泄露，被他人盗用导致额度耗尽或费用暴增 | 使用环境变量或 `.env` 文件，加入 `.gitignore` |
| 飞书 App Secret 提交到 Git | 机器人被恶意控制，发送垃圾消息 | 配置文件与代码分离，部署时注入 |
| 持仓 MD 文档包含真实资金量 | 隐私泄露，若仓库公开则风险极高 | MD 文档只记录股票代码和成本价，不记录总资金 |
| LLM Prompt 中注入未过滤的用户输入 | Prompt Injection，用户诱导机器人输出有害内容 | 对用户输入进行正则校验（只允许股票代码、中文简称） |
| 无日志脱敏 | 日志中记录 API Key、持仓详情，泄露风险 | 日志自动脱敏敏感字段，定期清理旧日志 |

## UX Pitfalls

| Pitfall | User Impact | Better Approach |
|---------|-------------|-----------------|
| 诊股响应超过 15 秒无反馈 | 用户以为机器人挂了，重复@导致请求堆积 | 先发送"正在分析中..."卡片，异步更新结果 |
| 晨报内容过多，手机端阅读困难 | 用户只看前3行，后面全部忽略 | 卡片顶部放"今日必读"摘要，详情可折叠 |
| 股票代码不支持中文简称 | 用户输入"中芯国际"无法识别，体验差 | 支持中文简称查询，建立代码-名称映射表 |
| 错误信息直接暴露技术细节 | 用户看到 `KeyError: 'close'` 等技术报错 | 所有错误转换为友好提示："数据暂不可用，请稍后重试" |
| 持仓股无事件时晨报仍长篇大论 | 用户觉得信息噪音大，逐渐忽略晨报 | 无重大事件时晨报精简为"持仓暂无重大变化" |
| 红标警示阈值固定 5% | 不同股票波动率不同，高波动股频繁误报 | 允许在 MD 文档中为每只股票自定义阈值 |

## "Looks Done But Isn't" Checklist

- [ ] **飞书连接：** 机器人能在群聊中响应@，但**未测试**网络断开后自动恢复 —— 验证：模拟网络中断30分钟，检查是否自动重连
- [ ] **晨报推送：** 本地能生成晨报，但**未测试**定时触发 —— 验证：将定时任务设为5分钟后，确认准时收到
- [ ] **数据获取：** 能获取单只股票数据，但**未测试**持仓组合（多只股票） —— 验证：持仓5只以上时数据完整性和响应时间
- [ ] **卡片格式：** 本地 JSON 验证通过，但**未测试**真实飞书渲染效果 —— 验证：发送到真实飞书群，检查手机端和 PC 端显示
- [ ] **节假日过滤：** 代码中有节假日判断逻辑，但**未验证**春节/国庆等长假 —— 验证：修改系统时间到休市日，确认任务跳过
- [ ] **LLM 输出：** 单次测试输出合理，但**未测试**连续20次是否稳定 —— 验证：对同一只股票连续查询，检查输出一致性
- [ ] **备用数据源：** 主数据源正常时备用源未测试 —— 验证：人为断开 Tushare，确认 AkShare/新浪财经降级生效
- [ ] **持仓解析：** 正常格式能解析，但**未测试**格式错误时的容错 —— 验证：故意破坏 MD 格式，检查错误提示是否友好

## Recovery Strategies

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| 飞书 WebSocket 僵尸连接 | LOW | 自动重连机制触发，或手动重启进程 |
| Tushare 额度耗尽 | LOW | 切换备用数据源，次日额度重置后恢复 |
| 消息卡片超限 | LOW | 自动裁剪内容后重发，或分片发送 |
| LLM 幻觉输出 | MEDIUM | 立即撤回/更正消息，检查 RAG 数据注入是否正常 |
| 定时任务静默失败 | MEDIUM | 检查 Cron/APScheduler 日志，手动补发晨报，修复时区/环境变量问题 |
| 持仓 MD 格式破坏 | LOW | 回退到 SQLite 缓存的持仓，通知用户修复 MD 格式 |
| 范围蔓延导致项目失控 | HIGH | 冻结功能开发，列出核心功能清单，砍掉非核心代码，重新聚焦 |
| 多 Agent 架构失控 | HIGH | 重构为单 Agent + Tools 架构，保留 Prompt 工程成果 |
