# 项目研究综合摘要

**项目:** 智能金融晨报与交互分析系统
**领域:** 个人智能金融助手 / 飞书机器人
**研究日期:** 2026-04-27
**置信度:** HIGH

---

## 执行摘要

本项目是一个面向 A 股/港股短线投资者的个人智能金融助手，核心能力为两大场景：每日开盘前（08:30）自动推送个性化金融晨报，以及用户在飞书群内 @机器人输入股票代码后 30 秒内获取 AI 诊股分析。基于研究，推荐采用**单进程、单线程（异步事件驱动）的 Python 应用架构**，遵循"能串行就不并行，能内存就不外存，能单文件就不分布式"的设计原则。这是个人 Demo，不是微服务。

技术栈选择围绕 Python 3.11+ 生态展开，飞书侧使用官方 `lark-oapi` SDK 的 WebSocket 长连接模式（无需公网 IP），数据层以 Tushare 为主、AKShare 为备用，AI 层采用 Claude 为主模型、Kimi 为降级备选。关键风险集中在三个方面：飞书 WebSocket 长连接的"假存活"问题、Tushare 免费额度耗尽导致数据中断、以及 LLM 金融幻觉可能输出虚假数据。这些风险均可通过双重健康检查、调用计数器自动降级、强制 RAG 架构（所有数据实时注入 Prompt）等策略有效规避。

## 关键发现

### 推荐技术栈

**核心运行时:** Python 3.11+ — 金融数据生态最丰富，Tushare/AKShare 均原生支持，asyncio 成熟。

**核心技术与版本:**

| 技术 | 版本 | 用途 | 推荐理由 |
|------|------|------|----------|
| Python | 3.11+ | 运行时 | 金融数据生态最丰富，asyncio 成熟 |
| lark-oapi | 1.5.5 | 飞书机器人 SDK | 官方维护，支持 WebSocket 长连接，自动处理 token 刷新和重连 |
| APScheduler | 3.11.2 | 定时任务调度 | Python 定时任务标准方案，cron 表达式支持完善，与 WebSocket 共存 |
| tushare | 1.4.29 | A 股基础数据（主） | 免费版覆盖日线/财务/基本面数据，DataFrame 输出，个人用量足够 |
| akshare | 1.18.11 | 备用数据源 | 完全免费无需注册，东方财富/新浪双源，Tushare 故障时自动降级 |
| peewee | 3.19.0 | SQLite ORM | 轻量级 ORM，单文件零配置，API 直观，代码量比 SQLAlchemy 少 50% |
| anthropic | 0.97.0 | Claude API 调用 | 官方 SDK，支持 streaming、tool use，分析质量公认最高 |
| openai | 1.99+ | Kimi API 调用 | Moonshot API 完全兼容 OpenAI 格式，复用同一 SDK 减少依赖 |
| pydantic-settings | 2.14.0 | 配置管理 | 内置 .env 加载 + 类型验证，2025 年 Python 配置管理标准 |
| loguru | 0.7.3 | 日志记录 | 零配置开箱即用，自动文件轮转/压缩，异常捕获装饰器 |
| httpx | 0.28.1 | HTTP 客户端 | 同步/异步双模式，HTTP/2 支持，requests 的现代替代 |

**明确不推荐:** Redis、Celery、PostgreSQL/MySQL、Django/FastAPI、LangChain、自建 Webhook 服务、飞书旧版卡片 JSON 1.0。

### 核心功能优先级

**P0 — 缺失则 Demo 无法演示核心流程:**
- 飞书机器人基础连接（WebSocket 长连接 + 消息监听）
- 个股代码查询响应（A 股 6 位代码 / 港股代码 / 中文简称识别，30 秒内返回）
- 单 Prompt AI 诊股（技术 + 基本面 + 资金面 + 情绪面四维分析，返回结构化卡片）
- 每日定时晨报推送（3 模块：隔夜全球市场 + 持仓公告摘要 + 今日板块前瞻，08:30 触发）
- Markdown 持仓读取（机器人每次主动读取更新上下文）
- 成本价 ±5% 红标警示（持仓股公告 + 价格双重触发，卡片红色高亮）
- AI 免责声明（每条消息底部固定添加）

**P1 — 增强体验，不阻塞核心:**
- 飞书交互卡片模板（非纯文本，信息密度高、视觉层次清晰）
- 晨报扩展至 6 模块（龙虎榜、关键事件日历、技术支撑位）
- 卡片交互按钮（"查看详情"、"换一只股票"等回调）
- 历史诊断记录（SQLite 本地存储，支持"查看上次诊断"）

**P2 — 个人 Demo 阶段过度设计，明确 defer:**
- 情绪面新闻爬虫（财联社/华尔街见闻关键词抓取）
- 板块轮动跟踪（长期记录板块热度变化）
- 多数据源融合（AKShare + Tushare + 新浪财经交叉验证）
- 诊股结果反馈学习（用户标记"准/不准"）
- 独立 Sub-Agent 并行架构（第一期串行调用足够）
- 实时 L2 行情推送、自动交易执行、全市场扫描/选股

### 架构要点

本项目采用**单进程、异步事件驱动**架构，核心组件如下：

1. **FeishuGateway** — 飞书连接网关：WebSocket 长连接管理（心跳、重连、事件监听）、消息解析（@机器人 / 股票代码 / 中文简称提取）、卡片发送（模板变量填充、大小预检、分片策略）。只负责"连接飞书、收发消息"，不处理业务逻辑。

2. **MasterAgent** — 中央调度器：意图路由（诊股 / 晨报 / 持仓查询 / 帮助）、上下文组装（持仓 MD + 用户查询 + 历史对话）、单 Prompt 多维度分析（技术/基本面/资金面/情绪面）、合规包装（免责声明前置、语气弱化、风险分级）。**关键决策：单 Agent 而非多 Agent**，第一期严格使用单 Agent + 多维度 Prompt，降低延迟和成本，避免输出矛盾。

3. **DataFetcher** — 数据获取与适配层：统一封装 TushareAdapter（主源）、AkshareAdapter（备用）、SinaAdapter（备用），对外提供"按股票代码获取全维度数据"的单一接口。包含数据校验层（OHLC 合法性、停牌检测、复权统一）和调用计数器（Tushare 额度监控、自动降级）。

4. **CardBuilder** — 卡片构建器：模板注册、变量渲染、大小预检（>25KB 触发裁剪/折叠/分片策略）。推荐使用飞书卡片搭建工具可视化设计后获取 `template_id`，代码中仅传递变量。

5. **Scheduler** — 定时任务调度：APScheduler BackgroundScheduler（与 WebSocket 共存）、交易日历过滤、晨报生成任务（独立线程执行不阻塞主循环）、心跳自检（每 5 分钟发送测试消息验证通路）。

6. **PortfolioReader** — 持仓文档读取器：每次查询时读取并解析 `portfolio.md`，提取 YAML frontmatter 和 Markdown 表格，格式校验失败时回退到 SQLite 缓存并告警。

**关键架构决策记录:**
- 单进程 vs 多进程 → 单进程（个人 Demo，无并发需求，简化部署）
- 单 Agent vs 多 Agent → 单 Agent + 多维度 Prompt（降低延迟和成本，避免输出矛盾）
- 每次读取 MD vs 缓存 → 每次读取（用户明确要求，确保上下文最新）
- 模板 ID vs 手写 JSON → 模板 ID（飞书卡片搭建工具可视化维护，代码只传变量）
- 异步 vs 同步 → 异步（asyncio，WebSocket 事件驱动，避免阻塞）

### 必须规避的陷阱（Top 5）

1. **飞书 WebSocket 长连接"假存活"** — 机器人看似在线实则收不到消息。
   - 规避：实现双重健康检查（WebSocket 层 Ping/Pong 超时检测 + 业务层每 5 分钟自检），指数退避重连，确认事件订阅已添加 `im.message.receive_v1`，应用必须发布版本。

2. **飞书消息卡片 30KB 超限导致晨报推送失败** — 错误码 `230025` 或 `19036`。
   - 规避：发送前预检查卡片大小（`len(json.dumps(card).encode('utf-8')) / 1024`），超过 25KB 自动触发内容裁剪策略（折叠次要内容、plain_text 替代 lark_md、分段发送）。

3. **Tushare 免费额度耗尽导致数据获取中断** — 免费用户每日约 100-500 次调用，无数据返回仍扣积分。
   - 规避：实现调用计数器，接近额度上限时自动切换备用数据源（AKShare、新浪财经）；所有股票代码规范化（6 位数字 + .SZ/.SH）；批量获取数据减少 API 调用；本地 SQLite 缓存历史数据。

4. **LLM 金融幻觉——编造财报数据、错误匹配股票代码** — 推理型模型幻觉率可达 14.3%。
   - 规避：**强制 RAG 架构**：所有财务数据、股价、公告必须通过 API 实时获取注入 Prompt；System Prompt 明确写入"如果数据不在提供的上下文中，回答'数据暂不可用'，绝不要编造"；数值校验层比对 LLM 输出与原始数据；股票代码锁定（全程使用代码而非名称引用）。

5. **个人 Demo 范围蔓延——从"晨报+诊股"膨胀为"量化交易平台"** — 金融数据领域诱惑极多，容易陷入完美主义拖延。
   - 规避：严格遵循 PROJECT.md 的 Out of Scope 清单；第一期只做 3 个晨报模块；设定硬截止日期（2 周内跑通核心流程）；每日自问"今天做的工作是否让核心场景更快更稳"。

## 对路线图的直接影响

### 建议阶段结构

基于组件依赖关系和功能优先级，建议按以下 9 个阶段构建：

#### Phase 1: 基础骨架
**理由:** 无依赖，所有后续阶段的基础。
**交付:** 环境变量配置、日志配置、Pydantic 数据模型、SQLite ORM 建表、基础测试框架。
**覆盖功能:** 无直接用户功能，但为所有功能奠定数据模型和配置基础。
**规避陷阱:** 无。
**研究标记:** 标准模式，无需额外研究。

#### Phase 2: 飞书连接
**理由:** 核心入口，不依赖任何业务组件，可最早验证。
**交付:** WebSocket 连接 + 消息解析 + 卡片发送能力，机器人能在群聊中响应 @消息。
**覆盖功能:** 飞书机器人基础连接、AI 免责声明（卡片模板固定 footer）。
**规避陷阱:** 飞书 WebSocket 僵尸连接（Pitfall 1）、消息卡片 30KB 超限（Pitfall 2）。
**研究标记:** 需要验证 WebSocket 重连机制和卡片大小预检。

#### Phase 3: 数据层
**理由:** 核心能力，诊股和晨报均依赖数据获取。
**交付:** Tushare 主数据源、AKShare 备用源、数据校验、额度监控、统一接口封装。
**覆盖功能:** 个股代码查询响应（基础行情数据）。
**规避陷阱:** Tushare 额度耗尽（Pitfall 3）、金融数据质量陷阱——停牌/复权/空值（Pitfall 4）。
**研究标记:** 需要验证多数据源降级切换逻辑和脏数据注入测试。

#### Phase 4: 持仓管理
**理由:** 个性化基础，不阻塞核心功能但增强晨报和诊股体验。
**交付:** portfolio.md 格式模板定义、MD 解析 + 校验 + SQLite 缓存回退。
**覆盖功能:** Markdown 持仓读取、成本价 ±5% 红标警示（数据基础）。
**规避陷阱:** 持仓 MD 文档与代码不同步（Pitfall 9）。
**研究标记:** 标准模式，格式校验逻辑明确。

#### Phase 5: LLM 层
**理由:** 智能核心，可与数据层并行开发。
**交付:** Claude/Kimi 统一封装、诊股/晨报 Prompt 模板、合规检查 + 免责声明。
**覆盖功能:** 单 Prompt AI 诊股（分析生成能力）。
**规避陷阱:** LLM 金融幻觉（Pitfall 5）、合规声明流于形式（Pitfall 10）、多 Agent 过度工程化（Pitfall 8）。
**研究标记:** 需要验证 Prompt 输出一致性和数值校验层有效性。

#### Phase 6: 中央调度（业务编排）
**理由:** 依赖 FeishuGateway、DataFetcher、PortfolioReader、LLM 全部就绪后才能构建。
**交付:** 意图路由 + 上下文组装 + 流程编排，@机器人 → 数据获取 → LLM 分析 → 卡片发送端到端跑通。
**覆盖功能:** 个股代码查询响应（完整流程）、单 Prompt AI 诊股（端到端）。
**规避陷阱:** 诊股响应时间超过 15 秒（需异步处理 + 先发送"分析中..."占位卡片）。
**研究标记:** 需要验证端到端响应时间和意图识别准确率。

#### Phase 7: 卡片构建（用户体验）
**理由:** 依赖 MasterAgent 输出数据结构确定后，才能设计卡片模板。
**交付:** 诊股卡片模板、晨报卡片模板、模板渲染 + 变量填充、SizeGuard 大小预检。
**覆盖功能:** 飞书交互卡片模板、单 Prompt AI 诊股（卡片展示）、成本价 ±5% 红标警示（视觉呈现）。
**规避陷阱:** 消息卡片 30KB 超限（Pitfall 2）。
**研究标记:** 需要验证卡片在飞书手机端和 PC 端的渲染效果。

#### Phase 8: 定时晨报（自动化）
**理由:** 依赖 Scheduler、DataFetcher、LLM、CardBuilder 全部就绪。
**交付:** APScheduler + 交易日历过滤 + 晨报生成任务（独立线程）+ 心跳自检。
**覆盖功能:** 每日定时晨报推送（3 模块：隔夜全球市场 + 持仓公告摘要 + 今日板块前瞻）。
**规避陷阱:** 定时任务静默失败（Pitfall 6）、Tushare 额度耗尽（Pitfall 3）、卡片超限（Pitfall 2）。
**研究标记:** 需要验证节假日跳过逻辑和定时触发准确性。

#### Phase 9: 集成与打磨
**理由:** 所有组件就绪后的系统集成和稳定性验证。
**交付:** main.py 应用入口、错误处理完善、日志与监控、端到端测试、"Looks Done But Isn't" 清单逐项验证。
**覆盖功能:** 全部 P0 功能完整可用。
**规避陷阱:** 范围蔓延（Pitfall 7）、所有技术债务在可接受范围内。
**研究标记:** 需要全面验证所有集成点和恢复策略。

### 阶段排序理由

- **FeishuGateway 最先构建:** 不依赖任何业务组件，可独立验证飞书连接，提供最早的可用反馈。
- **DataFetcher 与 LLM 层可并行:** 两者无直接依赖，DataFetcher 提供数据，LLM 层提供分析能力，可同步开发。
- **MasterAgent 最后构建:** 依赖 FeishuGateway、DataFetcher、PortfolioReader、LLM 全部就绪，是业务编排的核心。
- **Scheduler 在 MasterAgent 之后:** 晨报需要调用 MasterAgent 的完整分析流程，因此必须在中央调度就绪后构建。
- **范围蔓延防控贯穿全程:** 每个阶段开始前审查 Out of Scope 清单，任何新需求 48 小时冷却期后再决定是否纳入。

### 研究标记

**需要额外研究的阶段:**
- **Phase 2（飞书连接）:** WebSocket 重连机制和卡片大小预检需要实际测试验证。
- **Phase 3（数据层）:** 多数据源降级切换逻辑和脏数据注入测试需要实际数据验证。
- **Phase 5（LLM 层）:** Prompt 输出一致性和数值校验层有效性需要多次测试验证。
- **Phase 7（卡片构建）:** 卡片在飞书手机端和 PC 端的渲染效果需要真实环境验证。
- **Phase 8（定时晨报）:** 节假日跳过逻辑和定时触发准确性需要模拟测试。

**标准模式可跳过研究的阶段:**
- **Phase 1（基础骨架）:** 配置管理、日志、ORM 均为成熟模式，文档充分。
- **Phase 4（持仓管理）:** MD 解析和格式校验逻辑明确，无复杂集成。
- **Phase 6（中央调度）:** 意图路由和流程编排为常规设计模式，依赖前置组件验证。
- **Phase 9（集成与打磨）:** 系统集成测试为常规工程实践，无特殊研究需求。

## 待填补的空白

1. **飞书卡片模板设计:** 研究确定了"模板 ID + 变量填充"策略，但具体的诊股卡片和晨报卡片模板需要在 Phase 7 中通过飞书卡片搭建工具实际设计并获取 template_id。建议在 Phase 2 完成后即开始卡片原型设计，避免阻塞 Phase 7。

2. **Prompt 工程质量:** 单 Prompt 多维度分析的输出质量高度依赖 Prompt 设计。研究提供了 Prompt 框架，但实际效果需要在 Phase 5 中通过多次迭代测试验证，特别是数值校验层和合规检查的有效性。

3. **持仓公告数据源:** 研究提到"需对接公告数据源（东方财富/新浪财经爬虫）"，但具体的公告获取接口和解析逻辑尚未明确。建议在 Phase 3 中优先验证公告数据的可获取性，如不可行则调整晨报模块内容。

4. **真实用户反馈:** 所有功能优先级和 UX 设计均为研究者推导，未经过真实用户验证。建议在 Phase 9 后尽快进行内部试用，收集反馈用于 v1.x 迭代。

5. **Claude API 成本估算:** 研究未提供具体的 LLM 调用成本估算。个人 Demo 阶段成本可控，但如果诊股频率较高或晨报内容较长，需要监控实际用量并在 PROJECT.md 中更新预算约束。
